# Carality 汽车人格测试站设计文档

## 1. 项目概述

### 1.1 产品定位

Carality 是一个以汽车选购为目标的“人格测试驱动型选车网站”。

用户通过连续答题，得到一个四字母汽车人格结果，并基于该结果获得：

- 1 台最优推荐车型
- 3 台备选车型
- 对应的推荐理由
- 配套的购车指南内容

该产品不是偏娱乐化的 MBTI 站点，而是偏“选车决策辅助工具”。人格测试承担流量入口和传播载体的作用，核心业务价值是帮助用户更快缩小购车选择范围。

### 1.2 第一阶段目标

第一版只覆盖以下能力：

- 汽车人格测试答题流程
- 四字母汽车人格结果输出
- 基于车型数据库的车型推荐
- 购车指南展示
- 面向 SEO 和传播的结果页与内容页

第一版不做以下功能：

- 登录注册
- 在线支付
- 定位获取
- 经销商联动
- 短视频广告系统

这些能力在架构上需要预留扩展空间，但不进入首版交付范围。

### 1.3 商业模式

第一版采用“免费测试 + 广告/导流”的增长模式。

因此产品设计优先级为：

1. 降低答题门槛
2. 提高结果页传播性
3. 强化推荐车型的点击与后续浏览
4. 为后续广告位和供应商链接预留承接位置

## 2. 用户与核心流程

### 2.1 目标用户

第一版主要面向中国市场 10 万到 30 万预算区间的主流家用车潜在购车用户，典型特征包括：

- 想买第一台车的用户
- 家庭购车用户
- 对车型认知不系统、需要降低选择成本的用户
- 愿意通过测试型内容快速获得建议的内容消费用户

### 2.2 核心用户路径

1. 用户从首页、搜索入口、内容页或分享链接进入网站
2. 用户匿名开始测试，无需登录
3. 用户完成连续答题
4. 系统根据答案计算 8 维分数和四字母人格结果
5. 结果页输出人格说明、1 台最优车型和 3 台备选车型
6. 用户进一步查看车型详情与购车指南
7. 用户分享结果页或继续浏览站内内容

### 2.3 关键产品原则

- 首次测试不强制登录
- 结果页以“推荐车型”为主，“人格解释”为辅
- 推荐理由必须可解释，不能仅输出黑盒结果
- 题目围绕真实购车偏好设计，而不是抽象心理测试
- 数据结构和模块设计需支持未来增量扩展

## 3. 汽车人格体系

### 3.1 维度定义

网站采用固定 4 组对立维度，共 8 个字母：

- `P / E`：实用 Practical / 情绪 Emotional
- `S / Q`：省钱 Saving / 品质 Quality
- `C / D`：舒适 Comfort / 驾控 Driving
- `B / V`：品牌 Brand / 性价比 Value

### 3.2 结果编码规则

最终结果始终输出固定顺序的四字母编码：

1. `P` 或 `E`
2. `S` 或 `Q`
3. `C` 或 `D`
4. `B` 或 `V`

例如：

- `PSCV`
- `EQDB`
- `PQCB`
- `ESDV`

### 3.3 双轨结果模型

系统不能只存储字母结果，还必须保留每个维度的分数。首版采用双轨结果：

- 字母结果：用于展示、记忆和传播
- 维度分数：用于推荐算法、内容匹配和后续扩展

建议在业务层统一使用以下 8 个原始维度分数字段：

- `practical_score`
- `emotional_score`
- `saving_score`
- `quality_score`
- `comfort_score`
- `driving_score`
- `brand_score`
- `value_score`

字母结果由这 8 个分数经过对立维度比较后生成。

### 3.4 平局策略

若某组对立维度分值相等，首版应定义稳定的平局处理规则，避免结果摇摆。建议默认规则如下：

- `P/E` 平局时默认 `P`
- `S/Q` 平局时默认 `S`
- `C/D` 平局时默认 `C`
- `B/V` 平局时默认 `V`

该规则的业务含义是：首版倾向优先给出更保守、更大众化的推荐方向。

## 4. 题目系统设计

### 4.1 题目目标

题目系统的作用不是做传统人格学测评，而是采集用户真实的购车取向，以便把用户偏好映射到选车决策维度。

### 4.2 首版题目规模

首版建议使用 `12` 题。

原因：

- 少于 10 题会削弱结果可信度
- 超过 16 题会明显拉高流失率
- 12 题足以覆盖 8 个维度并形成较稳定的推荐分层

### 4.3 题目类型

题目应由以下四类构成：

- 用车场景题
- 决策偏好题
- 消费心理题
- 取舍对比题

### 4.4 选项设计原则

每题应包含 2 到 4 个选项。

每个选项可为一个或多个维度加分，允许采用“主维度强加分 + 次维度弱加分”的方式。例如：

- 主要倾向 `S`，次要倾向 `V`
- 主要倾向 `Q`，次要倾向 `B`

题目设计需遵守以下原则：

- 用购车语言表达，不用心理学语言表达
- 尽量避免用户一眼看出“正确答案”
- 保证每组对立维度都在多个题目里被重复验证
- 保留后续新增题目与动态题库能力

### 4.5 示例题目方向

- 你买车时最不能妥协的因素是什么
- 预算紧张时，你更接受配置减少还是品牌下调
- 一台车只要空间舒适，你是否接受驾驶乐趣一般
- 当两台车都够用时，你更倾向选喜欢的还是划算的
- 你更在意长期用车成本还是看起来更高级的质感

首版不在设计阶段固定完整题库文案，但实现阶段需要配套产出正式题库。

## 5. 车型推荐模型

### 5.1 推荐目标

系统推荐的目标不是输出“客观最强车型”，而是输出“更符合用户偏好的车型”。

因此推荐模型需要从“车型优劣判断”转向“用户偏好匹配”。

### 5.2 车型画像

每一台车型都需要有一套 8 维画像权重，用于表达该车型在以下维度上的倾向：

- 实用
- 情绪
- 省钱
- 品质
- 舒适
- 驾控
- 品牌
- 性价比

建议使用统一数值区间，例如 `0-100`。

### 5.3 匹配计算

首版采用可解释的规则评分模型，不引入复杂 AI。

推荐流程如下：

1. 根据答案生成用户 8 维分数
2. 根据对立维度比较得到四字母人格
3. 读取所有车型的 8 维画像权重
4. 计算用户偏好与车型画像的匹配分
5. 结合必要的业务规则做排序修正
6. 输出推荐结果和推荐理由

### 5.4 输出结构

结果页输出：

- 1 台最优推荐车型
- 3 台备选车型

最优推荐需要表达“最适合你”的判断，备选车型需要表达“接近但风格略有差异”的补充价值。

### 5.5 推荐理由生成

推荐理由必须基于维度差异生成，避免空泛文案。理由来源包括：

- 用户最高权重维度
- 车型与用户在关键维度上的高匹配点
- 车型与其他推荐项的差异化优势

示例表达方向：

- 你更重视省钱和性价比，这台车的综合用车成本与配置表现更匹配
- 你在舒适维度显著高于驾控维度，因此更适合强调空间与乘坐体验的车型
- 你对品牌和品质的倾向较高，因此系统保留了更稳妥的品牌向备选

### 5.6 业务修正规则

首版允许在基础匹配分外增加少量修正规则，例如：

- 同分时优先推荐市场认知更清晰的车型
- 同分时优先推荐更适合主流家用场景的车型
- 对信息不完整或画像可信度不足的车型降低排序

这些规则应放在独立推荐策略模块中，不能硬编码在页面层。

## 6. 车型库范围

### 6.1 首版范围

第一版车型库控制在 `20-30` 台精选车型。

覆盖范围：

- 中国主流家用车市场
- 预算区间 `10 万 - 30 万`
- 覆盖轿车、SUV、新能源、燃油、插混等主流选择

### 6.2 首版策略

采用“人工精选 + 人工标注画像”的方式构建车型库，不做自动抓取。

原因：

- 首版更需要推荐准确性，而不是车型数量
- 人工标注更容易形成可解释的推荐逻辑
- 后续扩容时可以逐步半自动化

### 6.3 车型信息结构

每台车至少包含以下信息：

- 基础信息：品牌、车系、车型名、指导价区间、能源类型、车身类型
- 展示信息：主图、亮点标签、推荐语、适合人群
- 业务信息：购车建议、注意事项、导流链接预留
- 画像信息：8 维权重
- 分类信息：预算段、车型类别、场景标签

## 7. 功能范围

### 7.1 首版功能

- 首页
- 答题流程页
- 结果页
- 推荐车型卡片与车型详情
- 购车指南内容页
- 基础埋点与分享访问统计
- 后台数据维护能力

### 7.2 首版不做

- 用户登录与历史存档
- 评论互动
- 会员系统
- 在线支付
- 城市定位
- 本地经销商实时数据接入
- 广告配置后台

### 7.3 后续扩展预留

- 登录后保存测试结果
- 深度报告与支付能力
- 按地区输出本地供应商链接
- 短视频广告位与站内导流位
- 更复杂的车型筛选器与选车工具
- 内容推荐与运营活动配置

## 8. 技术架构

### 8.1 技术选型

推荐技术栈如下：

- 前端与全栈框架：`Next.js`
- UI：`React + TypeScript + Tailwind CSS`
- 数据库：`PostgreSQL`
- ORM：`Prisma`
- 参数校验：`Zod`
- 部署：`Vercel`
- 数据库托管：`Neon` 或 `Supabase`
- 分析埋点：`Plausible` 或 `PostHog`

### 8.2 架构原则

第一版采用单体全栈架构，但内部按业务模块分层，避免未来功能新增时耦合失控。

必须遵守以下原则：

- 页面层只负责展示与交互
- 业务规则必须沉淀到领域模块
- 推荐算法独立于页面
- 数据访问通过仓储或服务封装，不直接散落在页面代码中
- 扩展功能通过模块新增接入，而不是改写主链路

### 8.3 推荐模块划分

建议按照以下业务边界拆分：

- `quiz`：题目、选项、答题流程、会话
- `personality`：维度分数计算、字母结果生成、人格解释
- `vehicle-catalog`：车型库、标签、车型信息
- `vehicle-recommendation`：推荐算法、排序、推荐理由生成
- `buying-guide`：购车指南内容管理与展示
- `content`：SEO 内容页与专题内容
- `analytics`：事件埋点与分享来源记录
- `integration`：后续登录、支付、地理位置、供应商接入扩展
- `ad-slot`：未来广告位与内容植入扩展

该划分符合开闭原则：新增能力优先通过新增模块或扩展接口接入，不破坏现有核心模块。

## 9. 数据模型

### 9.1 核心表

首版建议至少包含以下数据表：

- `quiz_questions`
- `quiz_options`
- `quiz_sessions`
- `quiz_answers`
- `personality_results`
- `vehicles`
- `vehicle_dimension_weights`
- `vehicle_recommendations`
- `buying_guides`
- `content_posts`
- `share_visits`
- `analytics_events`

### 9.2 关键关系

- 一个题目对应多个选项
- 一个答题会话对应多条答案
- 一个答题会话生成一个人格结果
- 一个结果可对应多条推荐车型记录
- 一台车型对应一条或多条购车指南内容

### 9.3 设计原则

- 题库、结果、车型库、推荐结果必须解耦
- 结果计算数据应可复算，不依赖页面缓存
- 车型画像权重应可独立维护
- 推荐结果应持久化，以便后续做分析与复盘

## 10. 页面结构建议

### 10.1 首页

目标：

- 说明产品价值
- 引导开始测试
- 建立“这是一个帮你选车的工具”的预期

首页核心内容建议：

- 主标题与副标题
- 测试入口按钮
- 汽车人格体系简介
- 示例结果展示
- 推荐车型案例
- 常见问题

### 10.2 答题页

目标：

- 保持连续作答体验
- 降低中途流失

答题页建议：

- 单题逐步切换
- 顶部进度提示
- 明确单选交互
- 支持返回上一题
- 自动保存当前会话状态

### 10.3 结果页

结果页优先级建议：

1. 最优推荐车型
2. 3 台备选车型
3. 推荐理由
4. 四字母人格结果说明
5. 购车建议与相关文章
6. 分享入口
7. 广告与导流位预留

### 10.4 内容页

内容页可承接搜索流量与结果页扩展阅读，例如：

- 某人格类型适合买什么车
- 预算区间选车指南
- 新能源与燃油对比
- 家用 SUV 推荐专题

## 11. 可扩展性设计

### 11.1 登录扩展

首版不做登录，但需要保证匿名测试结果未来可与用户账户绑定。

建议做法：

- `quiz_sessions` 使用匿名会话 ID
- 后续登录后可将会话归属到用户

### 11.2 支付扩展

支付能力应作为独立模块引入，不影响现有测试和推荐主流程。

未来可用于：

- 深度购车报告
- 一对一顾问服务
- 高级筛选与订阅内容

### 11.3 地理位置与供应商扩展

车型详情与结果页应预留区域化信息位，未来可按城市提供：

- 本地门店入口
- 当地供应商链接
- 区域活动信息

### 11.4 广告扩展

广告应作为可配置插槽接入，不写死在业务组件中。建议未来广告插槽分布于：

- 结果页中段
- 内容页段落间
- 推荐车型列表后

## 12. 风险与控制

### 12.1 风险

- 题目设计不合理会导致推荐可信度不足
- 车型画像权重过于主观会导致结果偏差
- 首版车型覆盖太少可能影响用户感知完整性
- 如果结果页人格解释过重，可能偏离“帮助选车”的核心价值

### 12.2 控制策略

- 题目和车型权重采用人工校准迭代
- 首版车型库选择认知度高的主流车型
- 推荐理由尽量具体，减少泛化表达
- 将推荐点击率、完成率、跳出率纳入埋点分析

## 13. 实施顺序建议

建议按以下顺序推进：

1. 完成数据库模型设计
2. 完成题目系统与维度映射设计
3. 完成首版车型库与画像权重录入
4. 实现答题流程与结果计算
5. 实现推荐算法与结果页
6. 实现购车指南与内容页
7. 接入基础埋点与分享访问统计
8. 优化 SEO、导流位与后续扩展点

## 14. 设计结论

Carality 第一版应被实现为一个以“汽车人格测试”为入口、以“车型推荐”为核心价值的单体全栈网站。

在技术上采用 `Next.js + PostgreSQL + Prisma`，在业务上采用“人格测试 + 精选车型库 + 可解释推荐算法”的组合，在结构上通过模块拆分和数据解耦满足未来对登录、支付、广告、定位和供应商链接的扩展需求。

该方案在当前阶段兼顾了：

- 你的前端主导开发能力
- 快速上线验证的效率
- 首版业务的聚焦度
- 后续功能增长的可持续性
