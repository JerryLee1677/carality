# Carality Personality-First Refresh Design

## 1. Goal

本次改版分两部分同步收敛：

1. 首页改成单屏、低干扰、强 CTA 的入口页。
2. 结果链路改成“先人格、后推荐”，先把汽车人格判断做稳，再把车型推荐降级为后续弱入口。

本轮不追求把车型推荐算法一并做对。本轮目标是先建立一个独立、统一、可解释的人格层。

## 2. Problems To Fix

### 2.1 首页问题

- 首页信息密度过高，首屏出现多组卡片、样例结果、示例车型、补充说明，用户容易失焦。
- 首页展示内容与后端真实流程不一致，例如首页仍在暗示少量题目和直接车型输出，但后端已经是更长的问答流程。
- 视觉层面当前使用大面积土黄、青绿、半透明卡片和仪表盘式元素，整体不够克制，页面气质偏杂。

### 2.2 算法问题

当前实现不是单一算法，而是以下机制的混合：

- 分支题目选择
- trait 累加
- 人格 profile 规则命中
- 车型硬约束筛选
- 车型加权排序
- 展示型人格码

这些机制使用了不同的语义体系，导致“数据库人格”“展示人格”“推荐结果”之间并不统一。

### 2.3 当前算法的主要局限性

- 题目选择是规则排序，不是真正的自适应测评。
- trait 采用裸累加，没有标准化、反向计分、常模、稳定度或置信度。
- 人格判断和车型推荐共用同一批中间信号，但输出逻辑不是同一人格体系。
- 所谓硬约束在无匹配车型时会退回全量车型池，约束失效。
- 推荐得分权重为手工设定，没有校准依据。
- 结果解释可能与推荐排序冲突。

## 3. Product Direction

### 3.1 首页方向

首页改成单屏 Hero，不再承担教育过多产品细节的任务，只负责：

- 让用户快速理解“这是一套帮你识别汽车人格的测试”
- 提供直接开始测试的主入口
- 提供弱次入口进入购车指南

首页不再展示：

- 信息卡片
- 示例人格码
- 示例主推荐车型
- “12 道题完成测试”“移动端优先”“主推荐 + 备选”等说明
- “为什么适合首版上线”之类的二级说明区

### 3.2 结果链路方向

结果页从“直接给具体车型”改成“先输出汽车人格”。

用户完成答题后，主结果页只返回：

- 主人格名称
- 一句核心判断
- 匹配度或稳定度
- 强解释文案
- 生活场景描述
- 用车习惯描述
- 优势
- 注意点
- 弱入口：查看适合你人格的车型方向

本轮不在主结果页中直接输出具体车型列表。

## 4. Reference Model From Mainstream Tests

结合主流测试网站，可归纳出四类常见结果机制：

### 4.1 类型型

先形成若干对立维度的连续结果，再映射为单一类型。

适用性：
适合 Carality 当前要做的“汽车人格 archetype”。

### 4.2 连续特质量表型

先输出多个独立维度分数，再附加解释，不强行把所有人塞进极少数固定类型。

适用性：
可作为本项目弱可视化维度的呈现方式。

### 4.3 多维累加后映射型

先累加多个维度，再根据 Top-N 维度解释用户画像或映射到某类目标。

适用性：
适合本项目人格层的内部实现。

### 4.4 真正自适应测评型

根据前序回答动态选题，目标是减少题量并提升结果稳定性。

适用性：
当前项目暂时不做严格 adaptive testing，只保留轻量分支逻辑。

## 5. Personality Layer Design

### 5.1 Core Principle

第一阶段只做人格判断，不做车型推荐。

人格层必须满足：

- 题目答案先汇总到统一人格维度
- 人格定义、展示文案、接口输出共用一套 archetype 配置
- 车型库变更不影响人格结果
- 车型推荐后续成为人格层之后的第二阶段能力

### 5.2 Unified Dimension Model

本轮统一为 4 组核心人格维度，作为汽车人格层的基础：

1. 务实导向 vs 表达导向
2. 成本敏感 vs 品质体验
3. 舒适家庭 vs 驾驶掌控
4. 稳定保守 vs 尝鲜科技

这 4 组维度用于：

- 生成主人格
- 生成弱可视化
- 生成人格解释文案

不直接用于本轮输出具体车型。

### 5.3 Personality Snapshot

答题完成后，后端先生成一个统一的 `dimensionSnapshot`，包含：

- 每个维度的倾向值
- 主导倾向
- 稳定度或匹配度

其中“匹配度/稳定度”用于结果页展示，语义定义为：

- 当前用户在该主人格上的归属强度与结果稳定性
- 不是与其他用户比较后的百分位

### 5.4 Archetype Mapping

`dimensionSnapshot` 生成后，再映射到单一 `主人格 archetype`。

每个 archetype 统一维护以下内容：

- `code`
- `name`
- `subtitle`
- `summary`
- `decisionStyle`
- `lifeScenes`
- `usageHabits`
- `strengths`
- `cautions`
- `imageUrl` 或 `imageKey`

当前中文 archetype 风格继续保留，类似：

- 务实省心型
- 科技驾趣型
- 成熟品质型
- 感性体面型

不再让英文字母码承担主要展示角色。

## 6. Result Page Design

### 6.1 Tone

结果页参考用户提供的 archetype 海报式结果结构，但整体语气更克制，避免玩梗感过强。

目标气质：

- 像一份汽车决策画像
- 有记忆点
- 但不过分娱乐化

### 6.2 Information Structure

结果页主结构如下：

1. 人格图片区
2. 主人格名称
3. 一句副标题
4. 匹配度/稳定度
5. 长解释
6. 你的优势
7. 你的注意点
8. 生活场景
9. 用车习惯
10. 弱入口：查看适合你人格的车型方向

### 6.3 Image Handling

人格图片作为可选资源处理：

- 有图片时展示人格插画或角色图
- 没有图片时展示占位容器
- 数据结构保留图片字段，后续支持人工补录每个人格的图片

## 7. Homepage Design

### 7.1 Layout

首页改为单屏 Hero：

- 顶部保留站点导航
- 中央区域只保留核心标题、简短说明、主 CTA、弱次 CTA
- 页面纵向高度尽量控制在一屏内完成主要认知

### 7.2 Content

首页建议文案语义：

- 强调“识别你的汽车人格”
- 强调“帮助你理解自己的购车偏好”
- 不提前承诺具体车型推荐结果

### 7.3 Visual Direction

视觉方向改为更克制的汽车编辑风：

- 减少大面积暖黄和青绿对撞
- 减少玻璃卡片和数据看板感
- 保留少量汽车产品气质，但不再使用重仪表盘式结构
- 强化留白、标题对比、按钮识别性

## 8. API And Data Contract

### 8.1 Complete Session Output

完成答题后，主结果接口返回以下结构：

```ts
type PersonalityResult = {
  sessionId: string;
  personality: {
    code: string;
    name: string;
    subtitle: string;
    summary: string;
    decisionStyle: string;
    lifeScenes: string[];
    usageHabits: string[];
    strengths: string[];
    cautions: string[];
    matchScore: number;
    dimensionSnapshot: Array<{
      key: string;
      label: string;
      value: number;
      directionLabel: string;
    }>;
    imageUrl: string | null;
  };
  recommendationEntry: {
    label: string;
    href: string;
  } | null;
};
```

### 8.2 Current Recommendation Output Change

本轮不再把具体车型数组作为主结果接口的默认输出。

如果系统内部仍需暂存推荐层能力，应改为：

- 不在主结果页消费
- 不在首页或结果页首屏展示
- 仅通过弱入口进入后续页面

## 9. Architecture Changes

### 9.1 Split The Current Mixed Flow

当前混合流程应拆为两层：

#### Layer 1: Personality Engine

负责：

- 汇总答题信号
- 计算统一人格维度
- 映射 archetype
- 输出解释文案结构

#### Layer 2: Recommendation Engine

暂不作为主链路结果输出。

后续负责：

- 基于人格层结果和用户硬条件做车型筛选
- 生成人格对应的车型方向或具体车型列表

### 9.2 Source Of Truth

必须收敛为单一人格源，不允许并行存在：

- 一套数据库人格规则
- 一套展示人格码
- 一套单独的前端展示映射

未来必须只有一套人格 archetype 定义作为 source of truth。

## 10. Non-Goals

本轮不处理：

- 车型推荐精排校准
- 推荐转化优化
- 个性化价格或地区服务
- 登录、商业化、广告体系
- 严格的自适应测评模型

## 11. Success Criteria

本轮完成后应达到：

1. 首页进入路径足够清晰，用户首屏不会被卡片和示例信息分散注意力。
2. 结果页主输出从“具体车型”切换为“汽车人格画像”。
3. 人格结果的命名、文案、匹配度、维度快照来自统一人格层。
4. 接口和页面中不再并行使用两套人格语义。
5. 车型推荐被降级为弱入口，等待后续单独校准。
